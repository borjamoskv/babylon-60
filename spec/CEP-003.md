# CEP-003: Epistemic Provider Interfaces (Drivers)

**Status:** Frozen Normative Draft
**Layer:** Provider ABI (C5-REAL)
**Authors:** CORTEX Autonomous Design Swarm

---

## 1. Abstract

Because the CORTEX microkernel is explicitly stripped of all semantic, inferential, and trust-evaluation logic (as per CEP-000), these capabilities must be injected into the system via **Epistemic Providers** (Drivers). This document defines the abstract interfaces and interaction contracts that external modules must implement to participate in the CORTEX ecosystem.

---

## 2. The Provider Paradigm

Providers operate in "User Space" and communicate with the microkernel via the **Epistemic Commit Protocol**. A Provider reads a `CommittedState` or `ValidatedState` from the kernel, performs its specialized logic, and proposes a new state transition.

Providers are stateless from the kernel's perspective. Their output is entirely deterministic relative to the state of the DAG at the time of their execution.

---

## 3. Core Interfaces

### 3.1 `InferenceProvider`
**Responsibility:** To generate new epistemic objects (`Assertion`, `Evidence`, `Constraint`) based on external sensory input or analysis of the existing graph.
*   **Input:** Current `EpistemicState`, External Stimuli (e.g., User prompt, LLM context).
*   **Output:** A `ProposedState` containing new nodes and `SupportRelations`.
*   *Note:* The InferenceProvider proposes structural changes but does not compute absolute truth.

### 3.2 `TrustProvider`
**Responsibility:** To compute the `TrustValue` of assertions within a given state based on the topology of `SupportRelations` and cryptographic signatures.
*   **Input:** A `Logical Commit` (A state lacking trust valuations).
*   **Output:** A `Trusted Commit` (The identical topological state, but with the `epistemic_valuations` map populated).
*   *Note:* Trust is highly subjective. Different TrustProviders (e.g., a Bayesian driver vs. a Zero-Knowledge driver) may produce entirely divergent valuations for the same graph topology.

### 3.3 `StorageProvider`
**Responsibility:** To persist the byte-streams of the DAG blocks (the underlying KV-store).
*   **Operations:** `put(hash, bytes)`, `get(hash) -> bytes`, `has(hash) -> bool`.

### 3.4 `IdentityProvider`
**Responsibility:** To verify the cryptographic signatures and identity of agents issuing state proposals.
*   **Operations:** `verify(signature, payload, pubkey) -> bool`, `resolve(identifier) -> Identity`.

### 3.5 `HashProvider`
**Responsibility:** To abstract the cryptographic hashing algorithm, allowing the kernel to remain isolated from specific crypto library implementations.
*   **Operations:** `hash(bytes) -> Hash` (Conforms to CEP-002 rules).

### 3.6 `CapabilityProvider`
**Responsibility:** To enforce authorization boundaries based on object namespaces and issuing identities.
*   **Operations:** `can_write(identity, namespace) -> bool`, `can_read(identity, namespace) -> bool`.
