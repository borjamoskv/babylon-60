# CEP-001: Core Epistemic Microkernel Architecture

**Status:** Frozen Normative Draft
**Layer:** Core Kernel (C5-REAL)
**Authors:** CORTEX Autonomous Design Swarm (borjamoskv / MOSKV-1 APEX)

---

## 1. Abstract

La propuesta **CEP-001** define la arquitectura fundamental del microkernel de CORTEX. Establece que el núcleo del sistema **no ejecuta razonamiento, inferencia ni tasación de la verdad**. Su única responsabilidad es actuar como garante criptográfico y estructural de los objetos epistemológicos, sus relaciones y la evolución histórica del conocimiento. El microkernel se diseña como un "Git epistémico" inmutable y verificable.

Todos los motores analíticos, LLMs, solvers lógicos y proveedores de confianza son relegados al rol de **Epistemic Drivers** (módulos externos conectables) que se comunican con el núcleo a través de un estricto Interfaz Binario de Aplicación (ABI).

---

## 2. Core Axioms & Kernel Responsibilities

El microkernel de CORTEX asume exclusivamente **cuatro responsabilidades primitivas e inmutables**:

1. **Immutable Objects (Data Layer):** Almacenamiento de entidades (Aserciones, Evidencias, Restricciones) indexadas por el hash criptográfico de su contenido (Content-Addressable Storage).
2. **Immutable Relations (Graph Layer):** Mantenimiento de la topología semántica mediante `SupportRelations` inmutables.
3. **Immutable States (Snapshot Layer):** Versionado del grafo global de conocimiento mediante un Merkle DAG histórico.
4. **Commit Validation (Boundary Layer):** El protocolo de certificación estructural (Gatekeeper) que previene la corrupción del DAG.

## 3. Core Object Model (Capa de Datos y Relaciones)

El conocimiento no se almacena como texto plano estocástico; se descompone en tipos atómicos inmutables.

### 3.1 EpistemicObject (Base)
Todo objeto que entra al kernel hereda de una primitiva inmutable, forzando la indexación por contenido.
* **ID:** `SHA3-256(canonical_serialization)`
* **Invariante:** Los objetos son *write-once, read-many*. No pueden modificarse, solo dereferenciarse en estados futuros.

### 3.2 Tipos Primitivos
* **`Assertion`**: Un reclamo lógico o fáctico sobre la realidad.
* **`Evidence`**: Un fragmento crudo de datos observables (documento, log, hash de imagen).
* **`Constraint`**: Una regla de invariancia lógica (ej. mutuamente excluyente).

### 3.3 Relational Model (SupportRelation)
Para preservar la inmutabilidad de los datos, la semántica de "cómo" se relacionan los objetos se extrae a una entidad separada.
* **Estructura:**
  * `evidence_id`: Puntero a la causa.
  * `assertion_id`: Puntero al efecto.
  * `relation_type`: Enum (`SUPPORTS`, `REFUTES`, `OBSERVES`, `DERIVED_FROM`, `CONTRADICTS`).

### 3.4 Extensibility (AnnotationSet)
Los metadatos se desacoplan del modelo semántico base para permitir firmas y namespaces independientes sin colisiones.

```text
Annotation {
  key: String,
  namespace: String,
  value: String,
  value_type: Enum (String, JSON, CBOR),
  issuer: Identifier,
  signature: Bytes
}

AnnotationSet: List<Annotation>
```

## 4. Normative Constraints: Equality & Serialization

### 4.1 Equality

La identidad de un objeto es estrictamente matemática.

* Two EpistemicObjects are structurally equal iff `canonical_serialize(A) == canonical_serialize(B)`
* Their `Identifier` **SHALL NOT** participate in equality evaluation.
* Their ContentHash **MUST** be computed exclusively from the canonical serialization (See CEP-002).

### 4.2 Canonical Ordering

Para garantizar que múltiples implementaciones del kernel produzcan exactamente el mismo DAG:

* Every unordered collection **MUST** be sorted according to the canonical byte ordering before serialization.
* Maps **MUST** be sorted by key (lexicographically by bytes).
* Sets **MUST** be sorted by the byte value of their canonical elements.

## 5. Core State Model (EpistemicState)

El estado del sistema es un nodo inmutable dentro de un Grafo Acíclico Dirigido (DAG).

* **`state_id`**: UIDv7 (Ordenamiento temporal y unicidad).
* **`parent_states`**: Punteros a estados previos (Soporta bifurcaciones y uniones).
* **`merkle_root`**: Raíz criptográfica de los sub-árboles que componen el estado.

### 5.1 State Machine Lifecycle

El ciclo de vida de un estado epistémico es estricto y unidireccional:

```text
[ Proposed ] → [ Validated ] → [ Committed ] → [ Archived ]
```

**Transiciones Ilegales (Must Panic):**
* `Committed` → `Proposed` (**MUST NOT** occur)
* `Archived` → `Validated` (**MUST NOT** occur)

## 6. Epistemic Commit Protocol (El ABI del Kernel)

El sistema opera bajo un modelo de **Propuesta y Certificación**.

### 6.1 El Flujo de Transición
```text
[ Estocasticidad / IA ]            [ Invariante Estructural ]
Inference Engine Driver ─────────> Epistemic Commit (Kernel Gate)
(Genera Propuesta de Estado)       │
                                   │ (Auditoría Estructural Pura)
                                   ├─> ¿Hashes calculados son correctos?
                                   ├─> ¿Dangling references? (Huérfanos)
                                   ├─> ¿El DAG es acíclico?
                                   │
                                   ▼
                             Committed State (Inmutable)
```

## 7. Kernel Error Model (Typed Panics)

Cualquier violación del contrato estructural aborta la transición. El kernel debe devolver errores tipados formales.

| Error Code | Precondition | Postcondition (Recovery) |
| :--- | :--- | :--- |
| `ERR_INVALID_HASH` | El hash declarado difiere del hash del `canonical_serialize()`. | Abort transition. No state mutated. |
| `ERR_UNKNOWN_PARENT` | Un `state_id` en `parent_states` no existe en el DAG local. | Trigger DAG sync/fetch protocol. |
| `ERR_CYCLE_DETECTED` | Una cadena de relaciones `DERIVED_FROM` forma un ciclo. | Abort transition. Reject cyclic topology. |
| `ERR_DANGLING_REFERENCE` | Referencia a `evidence_id` o `assertion_id` no encontrada. | Abort transition. Ensure objects exist. |
| `ERR_INVALID_SIGNATURE` | La firma de una `Annotation` o `Commit` es matemáticamente inválida. | Abort transition. Log security event. |
| `ERR_CAPABILITY_DENIED` | El Driver carece de autorización para mutar el namespace. | Abort transition. Notify Identity Provider. |

## 8. Conformance & Test Vectors

Para garantizar la interoperabilidad absoluta entre implementaciones del Kernel CORTEX, todas las implementaciones **DEBEN** pasar la **Conformance Test Suite** especificada en **CEP-004**. La suite define los vectores de prueba oficiales, incluyendo inputs, raíces Merkle esperadas y resultados de Commit.
