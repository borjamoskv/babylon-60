# Threat Model v1: CORTEX Persist

## Overview
This document outlines the v1 Threat Model for CORTEX Persist, identifying key assets, threat actors, trust boundaries, and mitigation strategies based on the STRIDE framework and OWASP LLM guidelines.

## 1. Assets
- **Cryptographic Keys**: Ed25519 private keys for signatures, AES-GCM keys for encryption at rest.
- **Master Ledger**: The SQLite database and hash chain containing the immutable history.
- **Agent Memory**: Context and facts generated by the Swarm.
- **Audit Bundles**: Exported ZIP files containing logs, proofs, and compliance mapping.

## 2. Threat Actors
- **Compromised LLM / Subagent**: An agent executing a malicious prompt injection or generating hallucinated/poisoned facts.
- **Malicious Insider / Tenant**: Attempting to perform cross-tenant data access or rewrite ledger history.
- **External Attacker**: Attempting to extract plaintext secrets, PII, or disrupt the persistence layer.

## 3. STRIDE Analysis (Key Risks)
- **Spoofing**: Agents forging `CORTEX-TAINT` metadata. Mitigation: Ed25519 signatures enforced at the `Persist-Guardian` level.
- **Tampering**: Altering SQLite state directly. Mitigation: Master Ledger hash chain and planned external Sigstore integration.
- **Repudiation**: Modifying logs to hide an action. Mitigation: RFC3161 Timestamping and append-only ledger design.
- **Information Disclosure**: PII or API keys saved in memory. Mitigation: Secret Scanner and PII Scanner (Phase 2) applied *before* the persistence saga.
- **Denial of Service**: Infinite generative loops. Mitigation: Kill Criteria (AX-047) and Saga timeouts.
- **Elevation of Privilege**: Bypassing the Write-Path Contract. Mitigation: Strict module isolation (`engine/` vs `cli/`) and Byzantine validation.

## 4. Planned Mitigations
As part of the roadmap, CORTEX Persist will implement:
- Envelope Encryption with KMS support.
- A "Memory Firewall" for prompt injection and risk scoring.
- Deterministic Tamper Detection Suites.
