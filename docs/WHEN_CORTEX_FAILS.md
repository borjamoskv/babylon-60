# When CORTEX Fails

> *"Trust is built on the explicit documentation of boundaries."*

CORTEX-Persist is designed as a strict, zero-trust execution manifold. It prioritizes the cryptographic integrity of the ledger over the availability of the agent. This means **CORTEX will intentionally fail, crash, or abort operations** to prevent corruption, probabilistic theater, or untraceable state mutations.

This document outlines the explicit failure modes of the system, why they happen, and how to recover.

---

## 1. Epistemic Failures (SAGA-1 Aborts)

These failures occur before data touches the database. They are the admission gates.

### `CausalClosureError`
- **When it happens:** An agent attempts to persist a conclusion, decision, or memory without supplying the underlying deductive chain (the `causal_graph`).
- **Why it happens:** CORTEX forbids "magic leaps" in logic. Generative outputs are conjecture until proven.
- **Recovery:** Provide the `justification` or `evidence` nodes in the `observe()` call.

### `EntropyDriftThresholdExceeded`
- **When it happens:** The `MetaArbiter` detects that the current execution trajectory diverges by more than the allowed threshold (`max_drift`) from the canonical path in the `DivergenceMap`.
- **Why it happens:** The agent is hallucinating or caught in a stochastic loop, moving away from known-good state vectors.
- **Recovery:** The `ExecutionControl` signal automatically returns `reroute` or `halt`. Developers should catch this and reset the agent context to the last safe checkpoint.

---

## 2. Storage & Ledger Boundary Failures

These failures occur at the physical database layer.

### `sqlite3.DatabaseError: not authorized`
- **When it happens:** Code attempts to execute `INSERT`, `UPDATE`, or `DELETE` directly against the database bypassing the `CortexEngine` runtime.
- **Why it happens:** The physical `sqlite3.set_authorizer` detects that the `CORTEX-TAINT` token is missing from the thread context. This prevents lateral SQL injections or bypasses.
- **Recovery:** Always route mutations through `CortexEngine.observe()` or the `@sovereign_persist` decorator.

### `MerkleChainMismatchError`
- **When it happens:** The daemon attempts to append a new block to the Ledger, but the `prev_hash` does not match the actual hash of the previous row.
- **Why it happens:** The database was tampered with outside of the CORTEX runtime (e.g., someone manually deleted a row in the SQLite file).
- **Recovery:** **Terminates the process instantly (SIGKILL behavior).** The ledger is compromised. You must restore from a known-good backup or run the `cortex ledger heal` CLI tool to fork the ledger from the last valid block.

### `AsynchronousOOMDrop`
- **When it happens:** The system is ingesting > 390k ops/sec and the asynchronous batch worker queue exceeds the available RAM before writing to the WAL.
- **Why it happens:** Thermal throttling, disk I/O stalls, or memory exhaustion. 
- **Recovery:** If using mission-critical mode, initialize the engine with `sync_flush=True`. This lowers throughput to ~50k ops/sec but guarantees ACID commit per observation without asynchronous queues.

## 3. Cryptographic Failures

### `Ed25519SignatureInvalid`
- **When it happens:** Verifying an Audit Pack JSON fails signature validation.
- **Why it happens:** The `cortex.crypto.verify` function detects that the JSON payload was altered after the Auth-Seal was generated by the CORTEX instance.
- **Recovery:** Reject the audit pack. The execution trace is invalid and mathematically proven to be tampered with.

### `MissingKeyringVaultError`
- **When it happens:** Starting CORTEX with `CORTEX_REQUIRE_SECURE_ENCLAVE=1`, but the OS keychain is inaccessible.
- **Why it happens:** Headless environments (like some CI runners or Docker containers) do not have dbus/keyring configured correctly.
- **Recovery:** Pass the master key securely via memory using `CORTEX_MASTER_KEY` environment variable, or configure a headless keyring backend (e.g., HashiCorp Vault plugin).
